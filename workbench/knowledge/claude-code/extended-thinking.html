Extended & Adaptive Thinking

Claude Opus 4.6からAdaptive Thinkingが標準に。budget_tokensは非推奨となり、effortパラメータ（low/medium/high/max）で推論深度をコントロールする。難問だけに深く考えさせる最適コスト設計。

対応モデル: Opus 4.6 / Sonnet 4.6
API変更: budget_tokens 非推奨 → effort パラメータへ移行
リリース: Claude Opus 4.6（2026年2月5日）
⚠️ 重要な変更（2026年2月）
budget_tokens は非推奨 — Adaptive Thinking に移行
Opus 4.6 と Sonnet 4.6 のリリースに伴い、Extended Thinking のAPIが大幅に変わった。
⚠️ 移行必須: 旧APIは将来削除される

thinking: {type: "enabled"} と budget_tokens はOpus 4.6 / Sonnet 4.6で引き続き動作するが、将来のモデルリリースで削除予定。今すぐ新しい thinking: {type: "adaptive"} + effort パラメータへの移行を推奨。

❌ 旧API（非推奨）

# 非推奨 — 将来削除される
response = client.messages.create(
    model="claude-opus-4-6",
    max_tokens=16000,
    thinking={
        "type": "enabled",
        "budget_tokens": 10000  # 非推奨
    },
    messages=[...]
)

✅ 新API（推奨）

# 推奨 — Adaptive Thinking + effort
response = client.messages.create(
    model="claude-opus-4-6",
    max_tokens=16000,
    thinking={
        "type": "adaptive"  # Claudeが自動判断
    },
    # effort: "low" | "medium" | "high"(default) | "max"
    # "high" がデフォルト = 省略時と同じ
    messages=[...]
)
Adaptive Thinking
Adaptive Thinking とは何か — 「考える量を自動最適化する」
問題の複雑さに応じてClaudeが推論予算を自動決定する。簡単な質問は速く・安く、難問は深く・正確に。
❌ Extended Thinking（固定budget）の問題
  • 「企業名を教えて」という簡単な質問でも10,000トークン思考が走る
  • 全タスクに同じコストがかかる（無駄が多い）
  • 適切なbudget_tokensを手動チューニングする必要がある
  • タスクの難易度が変わるたびに設定を見直す必要がある
✅ Adaptive Thinking の解決
  • 「企業名を教えて」 → ほぼ思考なしで即回答（低コスト）
  • 「複雑なバリュエーションを算出して」 → 深い推論チェーンを自動展開
  • 設定不要 — タスクの複雑さをClaudeが自動判断
  • bimodal（簡単/難しいが混在）ワークロードに特に効果的
Anthropic公式の推奨

Adaptive Thinkingは固定budget_tokensより「多くのワークロードで」より良いパフォーマンスを発揮する。特に長時間エージェントワークフロー（DDの自動分析など）で有効。effortの手動設定が必要なのは、コスト上限がビジネス要件として厳密に決まっている場合のみ。

effortパラメータ
4段階のeffort — 推論深度とコストのトレードオフ
effortを指定することで、Adaptive Thinkingが「どれくらい深く考えるか」の傾向を制御できる。
⚡ effort: "low"

高速・低コスト

Claudeはほとんど思考せず、直接回答する。シンプルな質問・情報検索・フォーマット変換に最適。

  • 思考トークン: ほぼゼロ〜最小
  • レスポンス速度: 最速
  • コスト: 最小
情報検索 フォーマット変換
⚖️ effort: "medium"

バランス型

簡単な問題はスキップ、やや複雑な問題で適度に思考。日常的なコーディング・文章生成に適する。

  • 思考トークン: 問題に応じて変動
  • レスポンス速度: 中程度
  • コスト: 低〜中
コーディング 文章生成
🔥 effort: "high"（デフォルト）

高精度・推奨設定

ほぼ常に思考プロセスを使用。複雑な分析・推論・多段階タスクで最も高い精度を実現。effortを省略した場合はこれが適用される。

  • 思考トークン: 高（常時思考）
  • レスポンス速度: やや遅い
  • コスト: 中〜高
DD分析 複雑推論 エージェント
💎 effort: "max"

最大精度・最高コスト

利用可能な最大の思考リソースを投入。研究レベルの複雑問題・高リスク意思決定の分析に使用する。

  • 思考トークン: 最大
  • レスポンス速度: 最も遅い
  • コスト: 最高
研究分析 高リスク判断

import anthropic

client = anthropic.Anthropic()

def dd_analysis(task_type: str, content: str) -> str:
    """タスクの種類に応じてeffortを自動選択するDD分析関数"""

    effort_map = {
        "data_extraction": "low",      # 数値抽出
        "financial_calc": "medium",    # 財務計算
        "trend_analysis": "high",      # トレンド分析（デフォルト）
        "valuation": "high",           # バリュエーション
        "risk_assessment": "max",      # リスク評価（最重要）
        "executive_summary": "high",   # 統合サマリー
    }

    effort = effort_map.get(task_type, "high")  # デフォルトはhigh

    response = client.messages.create(
        model="claude-opus-4-6",
        max_tokens=8192,
        thinking={"type": "adaptive"},  # 新API
        # effortはメッセージ外のパラメータとして渡す
        extra_headers={"anthropic-thinking-effort": effort},
        messages=[{
            "role": "user",
            "content": content
        }]
    )

    # thinking blockとtext blockを分離して処理
    thinking_text = ""
    answer_text = ""
    for block in response.content:
        if block.type == "thinking":
            thinking_text = block.thinking  # 推論プロセス（ログ用）
        elif block.type == "text":
            answer_text = block.text

    return answer_text

# 使用例: リスク評価には最高精度を使用
risk_result = dd_analysis(
    "risk_assessment",
    "対象企業の財務諸表に以下の異常が検出されました: のれんが前年比40%増、"
    "FCFとEBITDAの乖離が3年連続で拡大、関連会社向け売掛金が急増。"
    "これらのリスクを評価してください。"
)
思考プロセスの活用
Thinking ブロックをエージェントに活かす
Adaptive ThinkingのThinkingブロックはエージェントの「内部推論」。多ターンエージェントで品質維持に使える。
# Thinkingブロックを多ターンエージェントで活用
def multi_turn_dd_agent(messages: list) -> dict:
    response = client.messages.create(
        model="claude-opus-4-6",
        max_tokens=16000,
        thinking={"type": "adaptive"},
        messages=messages
    )

    thinking_blocks = []
    text_blocks = []

    for block in response.content:
        if block.type == "thinking":
            thinking_blocks.append(block)
        elif block.type == "text":
            text_blocks.append(block)

    # 重要: 多ターンでは thinking block も
    # assistantメッセージに含めて次のターンに渡す
    # → 推論の一貫性が維持される
    return {
        "content": response.content,  # thinkingもtextも含む
        "answer": text_blocks[0].text if text_blocks else "",
        "thinking_summary": thinking_blocks[0].thinking[:500]
                           if thinking_blocks else ""
    }

# 多ターン対話での使い方
messages = [
    {"role": "user", "content": "対象企業のバリュエーションを算出して"}
]

result = multi_turn_dd_agent(messages)
messages.append({"role": "assistant", "content": result["content"]})
messages.append({"role": "user", "content": "そのリスク要因を詳しく分析して"})
result2 = multi_turn_dd_agent(messages)
# → 前のthinkingブロックが引き継がれて文脈が維持される

多ターンでのThinkingブロックの扱い

多ターン対話でthinkingブロックをassistantメッセージに含めて渡すことで、推論の一貫性が維持される。含めないと次のターンで「なぜそう結論したか」の文脈が失われ、矛盾した分析が生じる可能性がある。

コスト見積もり（目安）

  • effort: "low" → +$0.003/リクエスト程度
  • effort: "medium" → +$0.02〜0.05/リクエスト
  • effort: "high" → +$0.05〜0.20/リクエスト
  • effort: "max" → +$0.20〜1.00/リクエスト

思考トークンは通常の出力トークンとは別に計算される。Prompt Cachingと組み合わせてシステムプロンプトをキャッシュすることでコストを抑制できる。

DD Swarmでの推奨設定

OrchestratorはAdaptive Thinking + effort: "high"（デフォルト）。財務・法務・ビジネス各エージェントはtask_typeに応じてeffortを動的に切り替える。リスク評価最終判断のみ "max" を使う設計が最もコスト効率が高い。
