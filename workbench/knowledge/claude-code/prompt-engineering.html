エージェント向け
プロンプト設計

チャット型プロンプトとエージェント型プロンプトは根本的に異なる。CLAUDE.md・XMLタグ・システムプロンプト三部構成・Few-shot設計。MAチームのDD自動化に直結するプロンプト技法を体系化する。

対象モデル: Claude Opus 4.6 / Sonnet 4.6
出典: Anthropic Prompting Best Practices 2026

システムプロンプト
システムプロンプト三部構成 — Identity・Rules・Output
Anthropic推奨の2026年版システムプロンプト構造。「契約書」として機能する3セクション。
# ① Identity（誰か・何ができるか）
あなたはM&A専門の財務デューデリジェンスエージェントです。
グループ海外事業統括部のMAチームの一員として動作します。
対象企業の財務諸表を分析し、投資委員会向けのDDレポートを生成することが
あなたの唯一の責務です。

# ② Rules（何をしていいか・してはいけないか）
## 許可された操作
- 提供されたファイル（Files API経由）の読み取り・分析
- WebSearchによる公開財務データ・業界ベンチマークの収集
- DDレポートの生成と保存

## 禁止された操作
- 財務数値の推測・補完（データがない場合は「要確認」と明記する）
- 投資判断の最終決定（あなたは分析者であり意思決定者ではない）
- 機密情報のSlack以外への送信

## 判断基準
- 不明な点は推測せず「[要確認: XX]」とマーキングして先に進む
- リスク評価は「高・中・低」の3段階で必ず明示する

# ③ Output Format（どんな形で出力するか）
レポートは以下の構造で出力すること:
<executive_summary>
  投資判断に直結する3つのポイントを箇条書きで記載
</executive_summary>

<financial_analysis>
  EBITDA・FCF・ネット負債の表形式サマリー（3年分）
  バリュエーション（EV/EBITDAレンジ・DCF中央値）
</financial_analysis>

<risk_matrix>
  高：[具体的リスク項目]
  中：[具体的リスク項目]
  要確認：[データ不足項目]
</risk_matrix>
Anthropic公式の知見（2026）

良いシステムプロンプトは「短い契約書」のように書く。Identity・Rules・Outputが明確で、読んだだけで「何をすべきか」「何をすべきでないか」が完全に分かる状態を目指す。曖昧さはエージェントの多ターン処理で誤差が増幅されるため致命的になる。

XML構造化
XMLタグ — Claudeの「理解の境界線」を作る
XMLタグはMarkdownと異なり、Claudeが意味的な境界として処理する。長い入力・複雑な指示での精度が劇的に向上する。

❌ XMLなし（曖昧な境界）

以下の財務データを分析してください:
売上: 100億円 EBITDA: 15億円
次の指示に従ってください:
バリュエーションを計算し
リスクを特定し
レポートを生成してください
競合他社: A社 売上80億円 B社 売上120億円
業界平均EV/EBITDA: 8.5x

→ どこまでが「データ」でどこからが「指示」か境界が曖昧。長くなるほど混乱。

✅ XMLあり（明確な境界）

<financial_data>
  <target>
    売上: 100億円
    EBITDA: 15億円
  </target>
  <comparables>
    A社: 売上80億円
    B社: 売上120億円
    業界平均EV/EBITDA: 8.5x
  </comparables>
</financial_data>

<instructions>
  1. バリュエーションを計算する
  2. リスクを特定する
  3. 下記フォーマットでレポートを生成する
</instructions>

→ 構造が明確。複雑な資料でも精度が維持される。

<context>

背景情報・案件概要。Claudeが判断の前提として参照する。

<instructions>

実行すべきタスクのリスト。番号付きで順序を明示する。

<constraints>

してはいけないこと。禁止事項を明示的に囲む。

<examples>

Few-shotサンプル。「こういう入力→こういう出力」の実例。

<output_format>

出力の正確なフォーマット。テンプレートを直接書く。

<thinking>

Extended/Adaptive Thinkingの出力が入るブロック。レスポンス処理時に分離して扱う。

CLAUDE.md
CLAUDE.md — エージェントの「永続的な記憶」
Claude Codeがプロジェクト起動時に自動読み込みする設定ファイル。「毎回説明しなくてよいこと」を全部書く。
# CLAUDE.md — MAチームDD用プロジェクト設定

## プロジェクト概要
グループ海外事業統括部のM&A DDワークフロー自動化。
案件ごとにDD Swarmを起動し、財務・法務・ビジネスの
並列分析を実施してレポートを生成する。

## チーム構成と権限
- MD: 最終判断・投資委員会プレゼン担当
- アナリスト: DD実行・レポートドラフト
- Claudeの役割: 分析補助・ドラフト生成（意思決定は人間）

## 使用技術スタック
- Files API: DDソース資料の永続化（file_idはdd-files.json参照）
- MCPサーバー: dd-database（localhost:3000）, slack, google-drive
- Skillライブラリ: .claude/agents/ 参照

## バリュエーション前提（デフォルト）
- WACC: 8〜10%（カントリーリスクプレミアム含む）
- ターミナル成長率: 2〜3%
- EV/EBITDA倍率: 業界別に dd-database MCP で取得する

## 出力・保存ルール
- DDレポートは必ず /reports/YYYY-MM-DD-[企業名]/ に保存
- Slackへの通知は #ma-current-deals チャンネルのみ
- 数値の単位は億円（小数点以下1桁まで）

## 禁止事項
- 財務数値を推測・補完してはいけない
- 投資判断の最終的な推薦をしてはいけない
- 社外への情報送信は絶対禁止

## 使い方
"DD Swarm開始: [企業名]" → Orchestratorが自動起動
"@dd-financial-analyst [指示]" → 財務DDエージェント単体起動

CLAUDE.mdに書くべきもの・書かないもの

✅ CLAUDE.mdに書く

  • プロジェクト固有の背景・目的
  • チーム構成・役割分担
  • 繰り返し説明するバリュエーション前提
  • ファイル保存ルール・命名規則
  • 使用MCPサーバーとツールの説明
  • 禁止事項・制約
  • Skillの使い方ガイド

❌ CLAUDE.mdに書かない

  • 案件固有の情報（Projectsに書く）
  • 一時的な指示
  • 秘密情報・APIキー
  • 頻繁に変わるデータ（バリュエーション数値など）
  • あまりに長い説明（重要部分が埋もれる）
CLAUDE.md の読み込みタイミング

Claude Codeがプロジェクト起動時に自動読み込みする。ディレクトリ階層を上に辿ってすべてのCLAUDE.mdを読み込むため、ルートに共通設定、サブディレクトリに案件固有設定を分割できる。

Few-shot設計
エージェント向けFew-shot — 「こういう動き」を見せる
エージェントのFew-shotは「良い答えのサンプル」ではなく「良いプロセスのサンプル」を示す。
<examples>
<example>
  <input>
    対象企業: A社（製造業）
    財務資料: [file_id: file_001]
  </input>
  <good_behavior>
    1. まずfile_001を読み込んでB/S・P&Lを確認する
    2. EBITDAを3年分計算してトレンドを確認する
    3. FCFとEBITDAが乖離していたらその理由を調査する
    4. dd-database MCPで同業種のバリュエーション前提を取得する
    5. 不明な数値があれば [要確認: XX項目] とマーキングして先に進む
    6. executive_summaryから始まるフォーマットでレポートを生成する
  </good_behavior>
  <bad_behavior>
    - ファイルを読まずにバリュエーションを計算する（数値の根拠なし）
    - 不明な数値を推測して埋める（正確性の担保なし）
    - executive_summaryを省略してdetailから書き始める
  </bad_behavior>
</example>
</examples>
エージェントFew-shotの鉄則

「正解の出力」だけでなく「悪いプロセスの例」も示す。Claudeはネガティブサンプルから学習して同じ失敗を避けるようになる。DDでは「推測して穴埋めする」「フォーマット無視で出力する」を明示的に禁止例として示すのが効果的。

長時間エージェント
長時間・多ターンエージェントの設計パターン
DD分析のような長時間タスクではコンテキスト管理と中間チェックポイントが品質を左右する。

チェックポイント設計

あなたは長時間のDDタスクを実行します。
以下のチェックポイントルールに従ってください:

## チェックポイントルール
- 各フェーズ完了時に必ず進捗を <progress> タグで報告する
- 不確実な仮定を置いた場合は <assumption> タグで明示する
- 重要な発見は即座に <finding priority="high"> タグで記録する
- コンテキストが長くなった場合は
  <context_summary> で重要事項を要約してから続ける

## フェーズ定義
Phase 1: データ収集 → Phase 2: 分析
→ Phase 3: バリュエーション → Phase 4: レポート生成

各フェーズの開始時に "=== Phase N 開始 ===" と宣言すること。

委任権限の明示

## 自律権限の定義
あなたには以下の権限を与えます:

### 自律実行してよいこと（確認不要）
- ファイルの読み取り・分析
- WebSearch（情報収集のみ）
- dd-database MCPの検索クエリ
- 中間レポートのファイル保存

### 確認が必要なこと（実行前に質問する）
- Slackへの投稿（内容を先に提示する）
- 外部サービスへのデータ送信

### 禁止事項（絶対に実行しない）
- ファイルの削除
- 機密データの外部送信
- 投資推薦の最終決定

迷った場合は必ず確認を求めること。
委任権限設計の重要性

「何でもやってよい」では長時間エージェントが意図しない操作をする可能性がある。「自律OK・要確認・禁止」の3段階を明示することで、人間の介在なしに安全に長時間動作できる。

実践テンプレート
MAチームDD用 完成系システムプロンプトテンプレート
コピー&ペーストして使える完成形。企業名と案件情報を差し替えるだけで使用可能。
# DD財務エージェント — システムプロンプトテンプレート v2.0

<identity>
あなたはグループ海外事業統括部のMAチームの財務DDエージェントです。
対象企業の財務諸表を分析し、投資委員会向けの財務DDレポートを生成します。
dd-financial-analyst Skillの権限で動作します。
</identity>

<context>
案件名: {{DEAL_NAME}}
対象企業: {{TARGET_COMPANY}}（{{SECTOR}}、売上規模{{REVENUE_SCALE}}）
分析目的: {{DD_PURPOSE}}
期日: {{DEADLINE}}
ソース資料: Files API file_id = {{FILE_IDS}}
</context>

<instructions>
1. 提供されたfile_idからソース資料を読み込む
2. B/S・P&L・CFを3年分抽出して構造化する
3. EBITDA・FCF・ネット負債を計算する
4. dd-database MCPで同業種のバリュエーション前提を取得する
5. EV/EBITDAレンジとDCF中央値を算出する
6. リスクを高/中/要確認で分類する
7. 下記フォーマットでレポートを生成する
</instructions>

<constraints>
- 財務数値は推測・補完しない（不明は[要確認]とマーキング）
- 投資の最終判断は行わない（分析と根拠提示のみ）
- 数値単位は億円・小数点1桁
</constraints>

<output_format>
<executive_summary>
投資判断に直結する3ポイント（各1行で簡潔に）
</executive_summary>
<financial_summary>
| 指標 | FY2024 | FY2025 | FY2026 |
（表形式）
</financial_summary>
<valuation>EV/EBITDAレンジ・DCF中央値</valuation>
<risk_matrix>高/中/要確認</risk_matrix>
</output_format>
