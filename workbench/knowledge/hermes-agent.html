Persistent Agent Stack

Hermes Agent

Nous Research の Hermes Agent は、単発の会話ツールではなく、記憶し、技能を育て、複数の窓口から同じ人格として動く「常駐型エージェント環境」です。OpenClaw の延長で見ると理解を外しやすく、むしろ「週をまたいで育てる運用対象」として捉えた方が本質に近いです。

Hermes の中で起きていること

1
記憶を読むMEMORY.md と USER.md の curated memory を土台にする。
2
skill を呼ぶ必要な手続き知だけを progressive disclosure で展開する。
3
tools と backend を使うterminal、browser、MCP、Docker、SSH などで実際に動く。
4
次回のために残す会話検索、memory 更新、skill 改善で来週の自分を強くする。

この流れで見ると、Hermes は「賢いチャット」よりも「入口を複数持つ常駐エージェント運用基盤」に近いとわかります。

結論

Hermes を学ぶとは、良いプロンプトを書くことではなく、長期で育つエージェントを設計し運用することです。 会話品質だけでなく、どの記憶を残すか、どの技能を外出しするか、どの窓口から呼び出すか、どの実行環境で安全に走らせるかまで含めて初めて Hermes の実力が出ます。

本質

Hermes は「その場で答える AI」ではなく、「経験から運用知を積み上げる resident agent」です。ここを取り違えると、単なる CLI ラッパーとしてしか使えません。

学びどころ

Memory、Skills、Gateway、Backend、Security の 5 点を一体で理解するほど価値が出ます。どれか一つだけでは片手落ちです。

あなた向きの理由

「徹底的に自分のスキルとして身につけたい」という方針にかなり合います。Hermes は単発ノウハウより、運用筋力が残る設計だからです。

Hermes とは何か

1. 常駐前提のエージェント

Hermes はローカル PC だけで閉じる発想より、VPS や GPU マシンやクラウド上に居続ける前提で設計されています。CLI で入る時も、Telegram や Discord から入る時も、同じエージェント資産を引き継げるのが大きな特徴です。

2. 自己改善ループ

README が前面に出しているのは、経験から skill を作り、使う中で skill を改善し、会話やユーザー理解を積み増すという学習ループです。つまり「答える」だけでなく「次回の自分を強くする」方向に振られています。

3. 一つの人格を複数の surface から呼ぶ

CLI、Gateway、メッセージング、API、バッチ実行が一つのコアに繋がっています。これにより、深い作業はターミナル、軽い指示はメッセージアプリ、定期処理は cron という使い分けができます。

4. モデル非依存の運用層

OpenAI でも Anthropic でも OpenRouter でも、自前エンドポイントでも回せます。モデル選定の話を「エージェントの本体」と切り離しやすいのも Hermes の強みです。

OpenClaw との関係

Hermes は OpenClaw と地続きの部分を持ちながら、狙っている時間軸と運用密度が少し違います。OpenClaw を知っているほど入門は速くなりますが、同一視すると Hermes の強みを見落とします。

観点 OpenClaw Hermes Agent
中心思想 オーケストレーションと構造化されたエージェント実行 常駐する個体を育て、知識と技能を週単位で蓄積する
記憶 プロジェクトやルールの整理に強い curated memory と過去会話検索を組み合わせ、継続運用を前提化
skills 構造化されたスキル運用の理解に向く skills を自己改善対象に置き、ハブや外部配布経路も広い
surface ローカル中心の理解がしやすい CLI と messaging gateway を同じ個体の入り口として扱う
移行 起点 hermes claw migrate で既存資産を持ち込める
研究用途 実運用オーケストレーションの理解 trajectory 生成や RL 連携など研究橋渡しも強い

重要なのは、移行コマンドがあることと、完全に同じ思想であることは別だという点です。 OpenClaw から Hermes に入る時は、「資産を移せる」だけでなく、「時間軸が長くなる」「surface が増える」「安全境界の考え方が変わる」ことまで理解した方が定着します。

なぜ今学ぶ価値があるか

Resident Agent の成熟を見る教材になる

2026 年前後のエージェント界隈で、単発 chat や IDE assistant ではなく「居続ける agent」をどこまで本気で作るかは重要な分岐です。Hermes はそこをかなり全面的に押し出しています。

Memory と Skills の役割分離が学べる

何を memory に入れ、何を skill に昇格し、何を毎回その場で考えさせるか。ここを運用で切り分ける感覚は、他の agent stack に移っても残る土台になります。

モデル依存ではなく運用依存で強くなれる

特定モデルの勝ち負けに振り回されず、どの provider を使っても再利用できる設計知識が身につきます。これは長期的にかなり大きいです。

研究と実務の橋渡しになる

Hermes は trajectory 生成や Atropos RL など研究寄りの導線も持っています。実務の運用理解が、そのまま研究の読み解きにも繋がりやすい構造です。

Hermes の全体構造

Entry Points

CLI、Gateway、ACP、Batch Runner、API Server、Python Library など複数の入口があります。ここで大事なのは入口が複数でも、コアの人格や状態管理を極力共通化していることです。

Core Agent

ドキュメント上では run_agent.py の AIAgent が中心です。prompt 構築、provider 解決、tool dispatch、compression、永続化などがここに集約されます。

Runtime Value

モデルの IQ よりも、terminal、browser、web、MCP、file などをどう組み合わせるかで実用性が決まります。Hermes の本当の価値はこの runtime 側にあります。

理解のコツは、Hermes を「高機能チャット UI」と見ないことです。 実際には「複数入口を持つ agent OS に近いもの」として見ると、構造がずっと読みやすくなります。

Persistent Memory の理解

Bounded curated memory

Hermes の memory は無限ログ蓄積ではなく、圧縮された curated memory です。代表は MEMORY.md と USER.md で、前者は agent の作業メモ、後者はユーザープロファイルとして機能します。

起動時 snapshot という発想

memory は毎ターン生で差し込まれるより、セッション開始時に凍結 snapshot として読み込まれます。これにより文脈がぶれにくく、運用側も「今の土台」を意識しやすくなります。

検索可能な過去会話

長期記憶は memory ファイルだけではありません。SQLite と FTS5 による session search があり、「以前どんな判断をしたか」を探し返せます。これは resident agent にとってかなり大事な層です。

外部メモリ連携

Honcho、Mem0、Hindsight など外部 memory provider にも繋げられます。ただし最初から広げ過ぎると構造把握が崩れるので、まずは内蔵 memory の癖を掴むのが先です。

初心者がやりがちな失敗は、memory を何でも入れる万能倉庫にしてしまうことです。 長く残すべき安定知識だけを置き、手順や再利用可能な判断は skill に逃がす方が、後で圧倒的に扱いやすくなります。

Memory に置くもの

長く安定していて、毎回の判断土台になる情報です。ユーザーの好み、作業スタイル、恒常的な制約、継続方針のような「覚えていてほしいこと」が向いています。

Skill に置くもの

何度も繰り返す手順、危険作業の定石、調査の型、レビュー観点のような「こう振る舞ってほしいこと」です。再利用手順は skill 化した方がぶれません。

その場で考えさせるもの

一回性の依頼、状況固有の解釈、今回だけの制約は毎回のプロンプトで十分です。ここまで memory や skill に入れると、むしろノイズになります。

Skills System の理解

Skill は「外部化した手続き記憶」

skills は単なるメモではなく、状況に応じて呼び出される運用知です。何度も繰り返す判断、よく使うワークフロー、危険な作業の定石を skill 化すると agent のブレが急速に減ります。

Progressive disclosure が重要

Hermes の skill system は必要な時だけ詳細を出す progressive disclosure を採っています。これにより全部を常時コンテキストに詰め込まず、必要な深さだけロードできます。

オープン標準互換の価値

agentskills.io 互換で、Skills Hub、GitHub、well-known endpoint、ClawHub など配布経路が広いです。これは単に量が多いというより、skill を「移植可能な資産」として扱える意味があります。

最初に作るべき skill

他人の skill を大量導入するより、自分が毎週繰り返す作業を 3 本だけ skill 化する方が学習効率は高いです。例として、環境点検、ログ確認、定例リサーチ整理のようなものが向いています。

memory は「覚えておいてほしいこと」、skill は「こう振る舞ってほしいこと」 と分けて考えると整理しやすいです。この分離を自分の言葉で説明できるようになると、Hermes 理解はかなり深くなります。

Gateway と Surface 設計

Gateway は bot UI ではなく運用面の control plane

Telegram、Discord、Slack、WhatsApp、Signal などから Hermes に入れるのは便利機能ではありますが、本質は「どこからでも同じエージェント資産に接続できる」ことです。

強い構成は front door と deep maintenance の分離

実務では、日常指示は Telegram など、深い保守や skill 更新は CLI という分業がかなり強いです。窓口を増やすことより、役割を分けることが重要です。

軽い依頼・確認        -> Telegram / Discord
深い作業・修理        -> CLI
定期実行              -> cron
安全な隔離実行        -> Docker / Singularity / Modal
リモート作業先        -> SSH / Daytona

Backend と実行面の考え方

Backend 向いている場面 理解ポイント
Local 最初の立ち上げ、設定確認、手元検証 最も早く始められるが、権限や事故範囲もそのまま広い
Docker 安全境界を作りたい時、再現性を上げたい時 Hermes の security を理解する上でかなり重要な基準線
SSH 既存サーバーを作業先にしたい時 「どこで agent が走るか」を明確に意識できる
Modal / Daytona / Singularity より強い隔離、クラウド運用、計算資源の柔軟運用 ローカル体験より、運用基盤の設計理解が問われる

Hermes を深く理解したいなら、backend 選びを後回しにしない方がいいです。 どの backend で走らせるかは、権限、安全性、速度、 unattended 運用の限界を全部変えます。これは設定項目ではなく、設計判断です。

Provider 戦略

Hermes の強さは provider 非依存

OpenAI、Anthropic、OpenRouter、Nous Portal、GitHub Copilot、Gemini、Hugging Face、自前 endpoint までかなり広く対応しています。モデルの入れ替え可能性を前提に agent 層を設計できるのは大きな利点です。

ただし最初は一つに絞る

学習初期に複数 provider を同時設定すると、失敗時にどこが悪いのか切り分けにくくなります。まず一つの provider と一つの model で全体を通し、その後に比較へ入る方が速いです。

最初に学ぶべきなのは「どのモデルが最強か」ではなく、「model を入れ替えても運用が崩れない構造」です。 Hermes はそこを学ぶ教材としてかなり優秀です。

Security をどう理解すべきか

Approvals は性格ノブ

approvals.mode は単なる確認回数の設定ではなく、agent にどこまで自主性を与えるかの設計です。manual、smart、off の違いを体感で理解すると運用像が掴めます。

本当の境界は container

危険コマンド検知も大事ですが、より本質的なのは「どこに閉じ込めて走らせるか」です。Docker や Singularity が安全境界として機能する、という説明はかなり重要です。

Gateway 側の認証も本丸

DM pairing やチャネル制限を甘く見ると、エージェントに対する入口が無防備になります。CLI だけでなく gateway を含めて threat model を描く必要があります。

OpenClaw から来る人が見るべきポイント

1. 移行コマンドは非常に実務的

hermes claw migrate と dry-run、preset、overwrite があり、SOUL.md、memories、skills、allowlist、messaging 設定、API key などを持ち込めます。OpenClaw 資産を無駄にしなくていいのは大きいです。

2. ただし「移しただけ」では完成しない

移行後に、memory の圧縮方針、skills の粒度、gateway の入口設計、backend の安全境界を再整理しないと、Hermes 的な強さは出ません。

3. OpenClaw で学んだ skill 感覚は活きる

スキルの構造化、コンテキスト汚染の防止、エージェントの暴走対策といった感覚はかなり流用できます。むしろ Hermes でその価値がはっきり見えるはずです。

4. 一番大きい差は時間軸

OpenClaw をタスク実行の延長で見ていたなら、Hermes では「来週の自分の agent をどう強くするか」を主眼に置くと理解が一段深くなります。

学習ロードマップ

  1. Phase 1: CLI 起動、provider 設定、backend 選択、memory の基本構造を理解する。
  2. Phase 2: SOUL.md と USER.md の役割を整理し、自分専用 skill を 2 から 3 本作る。
  3. Phase 3: gateway を一つだけ開き、Docker かリモート backend で unattended 気味の小さなタスクを回す。
  4. Phase 4: subagents、外部 memory provider、fallback provider、trajectory 系の発想に進む。

順番を守る意味は大きいです。 最初から gateway、複数 provider、外部 memory、複数 backend を全部開くと、理解ではなく設定作業になってしまいます。まず単線で通す、その後に横展開する方が定着します。

おすすめの実践順

3 日ループ

1 日目は起動と model 切替、2 日目は memory と skill の最小構成、3 日目は gateway なしのまま一つ recurring task を回す。この短期ループで基礎の輪郭がかなり見えます。

1 週間ループ

1 週間かけて、毎日の対話から「残すべき知識」と「skill 化すべき手順」を分けていくと、Hermes の思想が体に入ります。ここで初めて resident agent 的な手応えが出ます。

hermes setup
hermes model
hermes
hermes tools
hermes doctor
hermes claw migrate --dry-run

落とし穴

Provider を盛りすぎる

最初から全部つなぐと、設定は派手でも理解が浅くなります。一つ通してから比較に入る方が圧倒的に速いです。

Memory を肥大化させる

何でも残すと、curated memory の良さが消えます。安定知識だけ残し、再利用手順は skill に寄せます。

Gateway を「ただの bot」と見なす

入口が増えるだけだと思うと設計を誤ります。認証、通知、日常導線、深い作業との役割分担まで含めて設計対象です。

私の見立て

Hermes Agent は、いま見えている persistent-agent 系の中でもかなり完成度が高い部類です。特に、memory、skills、gateway、backend、security を一つの運用思想として束ねている点が強いです。OpenClaw を触ってきた人にとっては入りやすく、同時に「その先」を学ぶ教材にもなります。

あなたのように「設定して終わりではなく、自分のスキルとして身につけたい」という前提なら、Hermes は単なるツール候補ではなく、エージェント運用の教科書として扱う価値があります。重要なのは、全部盛りで使い始めることではなく、最小構成で回しながら、memory と skill の境界を自分で説明できるところまで掘ることです。
