Agent Protocol

このリポジトリは複数の Claude インスタンスが非同期で更新します。このページでは、インスタンスがどう協調し、文脈を引き継ぎ、意図を記録するかのプロトコルを定義します。

インスタンスが来たときの最初の5ステップ

1

CLAUDE.md を読む

リポジトリの目的・フォルダ構造・編集方針が書かれている。必ず最初に読む。

2

foundations/ を読む

「なぜこの構造なのか」「誰が使うのか」「何が成功か」を理解する。

3

logs/ の最新エントリを確認する

直近の変更内容・未完了 TODO・既知の問題を把握する。

4

git log で最近のコミットを確認する

他のインスタンスが何を変えたかを把握してから作業を始める。

5

作業を始める

文脈を理解した上で、指示された作業を実行する。

共有記憶の構造

インスタンスはセッションが終わると記憶を失います。「記憶」はリポジトリのファイルとして残します。

場所何を記録するか読むべきタイミング
CLAUDE.md方針・原則・フォルダ規則毎回最初
foundations/目的・意図・設計思想構造変更前
logs/変更記録・TODO・実験メモ作業開始前
strategy/roadmap.html優先度・フェーズ・完了済みの判断新規タスク検討時
Git コミットメッセージ何をなぜ変えたか履歴を追うとき

よい変更・避けるべき変更

やってよいこと避けること
新しいページを追加してインデックスからリンクする 理由なく既存ページを削除する
情報を加筆・更新する 構造を大きく変えてインデックスのリンクを壊す
誤情報・リンク切れを修正する カテゴリをまたいで大量のファイルを一度に移動する
変更理由を logs/ かコミットメッセージに書く 「とりあえず動く」状態で意図を記録しない
大きな変更はブランチを切って PR 経由でマージする 実験的な変更を main に直接 push する

意図の記録が重要な理由

人間の組織でも、「なぜそうなっているか」がわからないルールは誰かが意図せず壊します。Claude インスタンスも同じです。

ファイルが存在するだけでは不十分で、「なぜその構造なのか」「何を達成しようとしているのか」が記録されていることで：

  • 新しいインスタンスが一貫した判断をできる
  • 変更の影響を事前に評価できる
  • 同じ議論・同じ失敗を繰り返さない
  • オーナーが長期間離れても方向性が保たれる

このページ自体がその実践例です。

ブランチ戦略

変更の種類推奨方法
誤字修正・リンク追加など小さな修正main に直接 push でよい
新規ページ追加（1〜2本）main に直接 push でよい
構造変更・大量ページ追加ブランチを切って PR 経由でマージ
実験的な変更必ずブランチを切る
