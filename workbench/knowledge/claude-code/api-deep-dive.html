⚡ Advisor Tool — 2026年3月 Beta公開

エグゼキューター + アドバイザーの分業モデル

従来のエージェントは「高品質モデル1つが全タスクを処理」していた。Advisor Toolは エグゼキューター（高速・安価）が実行を担当し、 アドバイザー（高知性・高コスト）が必要な判断ポイントだけに介入する ハイブリッドアーキテクチャ。長時間タスクでコストを抑えながら品質を維持できる。

# Advisor Tool — Python SDK 実装例
import anthropic

client = anthropic.Anthropic()

# Advisor Tool の定義（エグゼキューターが使うツールとして登録）
advisor_tool = {
    "name": "ask_advisor",
    "description": "複雑な判断や戦略的分析が必要な場合にOpus 4.6アドバイザーに相談する",
    "input_schema": {
        "type": "object",
        "properties": {
            "question": {"type": "string", "description": "アドバイザーへの相談内容"},
            "context": {"type": "string", "description": "判断に必要な背景情報"}
        },
        "required": ["question"]
    }
}

def handle_advisor_call(question: str, context: str = "") -> str:
    """Opus 4.6をAdvisorとして呼び出す"""
    advisor_response = client.messages.create(
        model="claude-opus-4-6",
        max_tokens=1024,
        messages=[{
            "role": "user",
            "content": f"[M&A DD分析アドバイザー]\n\n背景: {context}\n\n質問: {question}"
        }]
    )
    return advisor_response.content[0].text

# エグゼキューター（Haiku）がAdvisor Toolを使って長時間タスクを実行
response = client.messages.create(
    model="claude-haiku-4-5-20251001",  # 高速・安価なエグゼキューター
    max_tokens=4096,
    tools=[advisor_tool],
    messages=[{
        "role": "user",
        "content": "以下の財務データを分析してDD要約レポートを作成してください。..."
    }]
)

☁️ Managed Agents — インフラ不要のエージェント本番化

「エージェントを動かすサーバー」が不要になる

従来、Claude APIでエージェントを本番稼動させるには、自前でサーバーを立て、 コンテキスト管理・再試行・タイムアウト処理・スケーリングを実装する必要があった。 Managed AgentsはこれをAnthropicのクラウドが引き受ける。 開発者はプロンプトとツールだけを定義すればよい。

  • エージェント定義（JSON）をAPIに登録するだけでクラウドコンテナが自動起動
  • 自動Prompt Cachingが有効化（長いシステムプロンプトを毎回送らなくてよい）
  • ツール呼び出し・再試行・エラーハンドリングをManaged環境が管理
  • Webhookで完了通知を受け取れる（非同期実行対応）
  • 実行ログ・コスト追跡がダッシュボードで確認可能

📁 Files API — セッションをまたぐファイル永続化

DDソース資料を一度アップすれば何度でも参照可能

従来はAPIリクエストのたびにファイルを送信していた。大量のPDF・Excelを扱うDD分析では これが帯域・コスト・レイテンシのボトルネックになっていた。 Files APIは一度アップロードしたファイルをfile_idで参照できる。 複数のエージェントが同じソース資料を共有するマルチエージェントDDで特に効果的。

  • 対応形式: PDF・テキスト・画像・スプレッドシート（Excel/CSV）
  • 最大ファイルサイズ: 32MB / 1ファイル、合計500MB / API key
  • 有効期限: デフォルト30日（設定変更可）
  • アクセス制御: API keyスコープで分離（チーム間の誤参照なし）
  • マルチエージェント共有: 複数のClaudeインスタンスが同一file_idを参照可
# Files API — アップロードと再利用
import anthropic

client = anthropic.Anthropic()

# 1. DDソース資料を一度だけアップロード
with open("target_company_financials.pdf", "rb") as f:
    uploaded_file = client.files.upload(
        file=("target_company_financials.pdf", f, "application/pdf"),
    )

file_id = uploaded_file.id
print(f"ファイルID: {file_id}")  # file_abc123xyz

# 2. 財務分析エージェントが参照（ファイル再送不要）
financial_analysis = client.messages.create(
    model="claude-sonnet-4-6",
    max_tokens=4096,
    messages=[{
        "role": "user",
        "content": [
            {
                "type": "document",
                "source": {
                    "type": "file",
                    "file_id": file_id  # アップロード済みIDを参照
                }
            },
            {
                "type": "text",
                "text": "この財務諸表からEBITDA・FCF・ネット負債を抽出してください"
            }
        ]
    }]
)

# 3. リスク分析エージェントも同じfile_idを使用（再アップ不要）
risk_analysis = client.messages.create(
    model="claude-opus-4-6",
    max_tokens=4096,
    messages=[{
        "role": "user",
        "content": [
            {"type": "document", "source": {"type": "file", "file_id": file_id}},
            {"type": "text", "text": "財務上の主要リスクを特定し、優先度をつけてください"}
        ]
    }]
)

# 4. ファイル一覧・削除
files = client.files.list()
client.files.delete(file_id)  # 不要になったら削除

機能 4
Prompt Caching GA TTL 1時間に延長
大きなシステムプロンプトやコンテキストをキャッシュ。繰り返し使うDD分析で90%コスト削減が可能。
🏎️ Prompt Caching — コスト最適化の切り札

「毎回同じ長文プロンプトを送る」問題を根本解決

大規模なエージェントシステムでは、詳細なシステムプロンプト・Few-shotサンプル・参照文書など 毎回送信するコンテキストが膨大になる。Prompt Cachingは一度処理したトークンを 最大1時間キャッシュし、以降のリクエストでキャッシュトークン分の入力コストを90%削減する。

  • キャッシュTTL: 最大1時間（以前は5分、今回の更新で大幅延長）
  • 最小キャッシュサイズ: 1,024トークン（Haikuは2,048）
  • キャッシュヒット: 入力コストの10%のみ請求
  • キャッシュミス: 通常の入力コスト + キャッシュ書き込みコスト（+25%）
  • 有効戦略: 変わらない部分（システムプロンプト・参考文書）を先頭に、変わる部分（ユーザー入力）を末尾に配置
# Prompt Caching 実装例
import anthropic

client = anthropic.Anthropic()

# キャッシュ対象: 変化しない大きなシステムプロンプト
system_prompt = """
[M&A DDエージェント — 詳細手順書]

## 財務分析フレームワーク
... (5,000トークン規模の詳細手順) ...

## リスク評価基準
... (DDチェックリスト、評価基準) ...

## レポートフォーマット
... (出力テンプレート詳細) ...
"""

# cache_control で明示的にキャッシュ指定
response = client.messages.create(
    model="claude-sonnet-4-6",
    max_tokens=4096,
    system=[{
        "type": "text",
        "text": system_prompt,
        "cache_control": {"type": "ephemeral"}  # TTL: 1時間
    }],
    messages=[{
        "role": "user",
        # ここだけが毎回変わる（キャッシュ対象外）
        "content": "対象企業Aの財務DDを開始してください"
    }]
)

🧠 Thinking — Extended から Adaptive へ

推論予算の自動最適化

Extended Thinkingは推論バジェット（thinking tokens）を手動設定する必要があり、 「全タスクに最大トークンを使う」か「全タスクに最小トークンを使う」かの二択になりがちだった。 Adaptive Thinking（Opus 4.6）はモデルが問題の複雑さを評価し、 ターンごとに推論予算を自動決定する。

  • 簡単なタスク: Thinkingトークンをほぼ使わず高速・安価に応答
  • 複雑なタスク: 深い推論チェーンを自動展開して精度を最大化
  • Opus 4.6ではデフォルトでAdaptive Thinkingが有効（明示設定不要）
  • Sonnet 4.6: Extended Thinking（手動設定）は引き続き利用可能

response = client.messages.create(

🤖 Agent SDK — マルチエージェントの標準基盤

エージェントシステムの設計パターンを標準化

Claude Code の Agent Teams（実験的）とは異なり、Claude Agent SDK は 本番APIで使うマルチエージェントシステムをコードで構築するための公式ライブラリ。 オーケストレーター・サブエージェント・ハンドオフ・状態共有などのパターンが組み込まれている。

  • Orchestrator: 複数エージェントへのタスク分配と結果統合
  • Agent: ツール付きの単一実行ユニット（Skills定義可）
  • Handoff: エージェント間での制御権移譲（P2P通信）
  • SharedContext: エージェント間でのメモリ・状態共有
  • TaskList: 共有タスクキュー（Agent Teamsと同等の概念）
  • Python / TypeScript SDK両対応
# Claude Agent SDK — MAチームDD Swarmの実装例
from anthropic.agents import Orchestrator, Agent, SharedContext

# 共有コンテキスト（全エージェントがアクセス可能）
shared_ctx = SharedContext({
    "target_company": "株式会社X",
    "deal_size": "50億円",
    "dd_files": [financial_file_id, legal_file_id, business_file_id]
})

# サブエージェント定義（Skillsパターン）
financial_agent = Agent(
    name="財務DDエージェント",
    model="claude-sonnet-4-6",
    system_prompt="M&Aの財務デューデリジェンス専門家。財務諸表分析・バリュエーション・FCF分析を担当。",
    tools=[financial_analysis_tool, valuation_tool],
    context=shared_ctx
)

legal_agent = Agent(
    name="法務DDエージェント",
    model="claude-sonnet-4-6",
    system_prompt="M&Aの法務デューデリジェンス専門家。契約書・コンプライアンス・訴訟リスクを担当。",
    tools=[contract_review_tool, compliance_check_tool],
    context=shared_ctx
)

business_agent = Agent(
    name="ビジネスDDエージェント",
    model="claude-opus-4-6",  # ビジネス判断はOpusを使用
    system_prompt="M&Aのビジネスデューデリジェンス専門家。市場分析・競合調査・シナジー評価を担当。",
    tools=[market_research_tool, competitor_analysis_tool],
    context=shared_ctx
)

report_agent = Agent(
    name="レポート統合エージェント",
    model="claude-opus-4-6",
    system_prompt="各エージェントの分析結果を統合し、エグゼクティブサマリーを作成する。",
    context=shared_ctx
)

# オーケストレーターが全体を指揮
orchestrator = Orchestrator(
    agents=[financial_agent, legal_agent, business_agent, report_agent],
    strategy="parallel_then_synthesize"  # 並列実行→統合
)

# DD Swarm 起動
result = await orchestrator.run(
    "対象企業Xの総合DDを実施し、投資委員会向けレポートを生成してください"
)
print(result.final_output)

重要な視点 — APIレベルの理解がゲームチェンジャーになる理由

Claude Codeのチャット画面やAgent Teamsは「UIレイヤー」に過ぎない。APIレベルで機能を組み合わせることで、DDS自動化・日次モニタリング・投資委員会レポート生成を人間の介在なしに動かすシステムが構築できる。これが「1人の社員が1,000社を並列監視する」という新しいMAチームの姿に直結する。
