結論

Primary

Claude を外出先の主系統にする

スマホから投げる前提なら、現時点では Claude 系の方が運用の相性が良いです。

  • 短い指示で継続更新しやすい
  • `CLAUDE.md` を読ませる運用と噛み合う
  • このハブの「留守中の更新係」に向いている
Secondary

Codex は補助線として十分使える

OpenAI 側も web / ChatGPT モバイル / Slack に広がっているので、外部運用の候補には入ります。

  • Codex web でクラウド実行
  • ChatGPT モバイル経由で継続
  • Slack 連携で軽いタスク依頼
Fallback

重い整理は家に戻ってからやる

大規模な移設、構造変更、CSS 改修、複数ページの再編はローカル作業を基本にします。

  • Codex CLI / app / IDE
  • このローカル環境の Codex
  • 差分確認とテストを伴う作業

このハブで外から投げるべきタスク

外でやるべきもの

  1. 新技術の要約を `research` に追加する
  2. 比較ページや運用メモを `workbench` に足す
  3. 優先順位や次アクションを `strategy` `logs` に更新する
  4. 壊れたリンクや明らかな文言ミスを直す

家でやるべきもの

  1. フォルダ移設や大規模リネーム
  2. 全体デザインの刷新
  3. 複数セクション横断の構造改修
  4. GitHub 公開前の最終確認とコミット整理

運用ルール

この `claude` ハブは「公開できる技術ハブ」であり、「理念や作品そのもの」は別管理です。外から触らせるのは主にここで、ALIGN_Lab 本体は必要に応じて後から取り込む方針が安全です。

  • 変更依頼はカテゴリ単位で出す。例: `workbench/knowledge だけ更新`
  • 必ず目的を書く。例: `外出先からの運用方法を1ページに整理`
  • 壊したくないページは明示する。例: `index.html の構造は維持`
  • 変更後は `logs` に要点を残す
  • 公開前はリンク確認と差分確認を行う

このあと追加すると強いもの

Issue テンプレート 更新チェックリスト 公開前レビュー手順 Claude / Codex 比較表

次に整備するなら、まず `logs/change-log` と `remote-task-template` です。 外出中の依頼文を型化しておくと、Claude でも Codex でもブレがかなり減ります。
