Sub-agents & Agent Teams

1人のユーザーが∞のエージェントチームを指揮する。並列実行・P2P通信・共有タスクリストで、複雑なタスクを根本から変える。

リリース: 2026年2月5日
対応: Opus 4.6 / Sonnet 4.6
設定: 実験的フラグ必要
基本概念
Sub-agents と Agent Teams — 何が違うか
どちらも「複数のAIを使う」概念だが、通信アーキテクチャが根本的に異なる。
⬡ Sub-agents（サブエージェント）
  • メインエージェントが子エージェントを生成・指示
  • サブエージェント同士は直接通信できない
  • 全コミュニケーションはメインエージェント経由
  • シングルセッション内で動作
  • @エージェント名 でメンション（v2.1.90〜）
  • 独立したタスク（並列調査・並列テストなど）に最適
✦ Agent Teams（チーム型）
  • 独立したセッションとして各エージェントが動作
  • P2Pメールボックス通信：エージェント間で直接メッセージ
  • 共有タスクリストでチーム全体の進捗を管理
  • 各エージェントが独自のコンテキストウィンドウを持つ
  • チームリーダーが全体を統括、メンバーは専門タスクを担当
  • 成果物のレビュー・相互フィードバックが必要なタスクに最適
💡 使い分けの原則

Sub-agents = 作業が独立していてお互いの結果を必要としない場合（バックエンド調査 / フロントエンド調査 / テストカバレッジ確認を同時実行など）
Agent Teams = 一方の成果を他方がレビュー・修正・発展させる必要がある場合（仕様作成→レビュー→テスト実装の連携など）

セットアップ
Agent Teams の有効化
実験的機能のため、settings.json への1行追加が必要。

① settings.json に環境変数を追加

{
  "env": {
    "CLAUDE_CODE_EXPERIMENTAL_AGENT_TEAMS": "1"
  }
}

場所: ~/.claude/settings.json（グローバル）または プロジェクトルートの .claude/settings.json

② チームサイズの設定（オプション）

# 2〜16 エージェントのチームを組める
# デフォルト: 自動（タスク複雑度に応じて決定）
# 明示的に指定する場合はプロンプトで「5人チームで」と伝える

③ tmux（推奨・任意）

# tmux で各エージェントの進捗を並列ウィンドウで確認できる
tmux new-session -d -s agents
# 各ペインでエージェントの出力をリアルタイム監視
💡 必要なプランと条件

Pro または Max プラン（Opus 4.6 へのアクセスが必要）。Claude Code v2.1.80 以降。

Sub-agents
カスタムサブエージェントの定義と @ メンション
専門能力を持つサブエージェントを定義ファイルで作成できる。

サブエージェント定義ファイル

.claude/agents/dd-researcher.md
---
name: dd-researcher
description: M&A DDの初期調査を専門とするエージェント。
  企業概要・財務・リスクを並列で調査する。
tools: WebSearch, Read, Write
---

あなたはM&Aデューデリジェンスの専門家です。
以下の観点で調査を行い、構造化されたレポートを作成してください：
1. 企業概要（事業・代表・設立・グループ）
2. 財務状況（直近3年の売上・利益率）
3. リスク項目（法的・地政学・コンプライアンス）
4. 出典を必ず明記すること
言語：日本語で出力

使い方（@ メンション）

# メインエージェントから指示
@dd-researcher ターゲット企業「XX社」の
初期調査を実施してください。

# 結果を受け取って次のエージェントに渡す
@synergy-calculator 上記の調査結果を元に
SBIグループとのシナジーを試算してください。
📌 ポイント

定義ファイルは .claude/agents/ 以下に置く。tools で使えるツールを制限することで安全性を確保できる。

実践パターン
Agent Teams の典型的な構成
チームリーダーがタスクを分解し、専門エージェントが協調して完了する。

🔍 DDスウォームパターン（MAチーム推奨）

ターゲット企業の初期DDを複数エージェントが並列実行。最後に統合エージェントがレポートを生成。

  • Leader：タスク分解・進捗管理・最終統合
  • Agent A：企業概要・代表者・グループ構成
  • Agent B：財務分析・直近3年・業界比較
  • Agent C：リスクスキャン（法的・地政学・コンプライアンス）
  • Agent D：シナジー試算（SBIグループ視点）
  • Agent E：全結果を統合してディールメモ生成
実績

4人アナリスト×2日分の作業 → 約5分で完了（Anthropic社内実測）

📝 レビューループパターン

Writerが作成し、ReviewerがチェックしてWriterにフィードバック。人間が介入しないレビューサイクル。

  • Leader：要件定義・完了基準を設定
  • Writer：ドキュメント / コード / レポートを作成
  • Reviewer：品質チェック・矛盾検出・改善提案
  • Writer：フィードバックを受けて修正
  • Leader：完成品を最終確認
Agent Teamsが必要な理由

ReviewerがWriterに直接フィードバックを送れる（P2Pメールボックス）。Sub-agentsだけでは実現できないパターン。

⚡ 並列コードレビュー

  • Security担当：セキュリティ脆弱性のチェック
  • Performance担当：パフォーマンス問題の検出
  • Style担当：コーディング規約・可読性
  • Leader：全結果をPRコメントにまとめる

📊 マルチソース調査

  • Agent A：Bloomberg / Reuters検索
  • Agent B：現地語メディア検索
  • Agent C：規制・法令データベース
  • Leader：統合レポート作成

🧪 並列テスト実行

  • Agent A：ユニットテスト実行・修正
  • Agent B：統合テスト実行・修正
  • Agent C：E2Eテスト実行・修正
  • Leader：全テスト通過を確認

✅ やるべきこと

  • 完了基準を最初に明確化する：「何をもって完了とするか」をLeaderに伝える
  • エージェントごとに専門を絞る：1エージェント1役割の原則を守る
  • 共有タスクリストを活用する：進捗の可視化でLeaderの管理コストを下げる
  • 定期的に監視する：放置しすぎると軌道が逸れる。特に最初の3分は要確認
  • 小さなチームから始める：2〜3エージェントで動作確認してから規模を拡大
  • Skills と組み合わせる：専門知識はSkillとして定義し、エージェントに読み込ませる
  • tmuxで並列監視する：各エージェントの出力をリアルタイムで確認できる

⚠️ 避けるべきこと

  • 長時間放置しない：エージェントは30分以上放置すると方向がずれやすい
  • 16エージェントを超えない：現状の上限は16。それ以上は不安定になる
  • 曖昧なタスクを与えない：「調査して」より「○○社の財務リスクを高/中/低で分類して」の方が良い
  • 全エージェントに全ツールを与えない：不要なツールアクセスは安全リスクになる
  • Sub-agentsとAgent Teamsを混同しない：P2P通信が不要なら Sub-agentsで十分
  • 機密情報を無制限に渡さない：エージェントが参照するデータの範囲を明示的に制限する
実装例
MAチーム向け：DDスウォームの実際のプロンプト
コピーして使えるプロンプトテンプレート。

メインエージェントへの指示（日本語）

Agent Teamsモードで以下のDD初期調査を実施してください。

【ターゲット企業】: [企業名を入力]
【調査目的】: SBIグループによる買収可能性の初期評価
【完了基準】: 5ページ以内のディールメモ（日本語）

チーム構成：
- Agent A（企業概要担当）：事業内容・代表者・設立・グループ構成・主要拠点
- Agent B（財務担当）：直近3年の売上・利益率・負債・業界平均との比較
- Agent C（リスク担当）：ネガティブニュース・訴訟・コンプライアンスリスク・地政学
- Agent D（シナジー担当）：SBIグループとのコスト削減・クロスセル・技術統合シナジー試算
- Agent E（統合担当）：A〜Dの結果を受け取り、ディールメモに統合

各エージェントは作業完了後、Agent Eに結果を送信してください。
Agent Eは全結果が揃い次第、社内フォーマットでディールメモを作成してください。

ユースケース推奨理由
DD初期調査（企業概要・財務・リスクを並列実行）Agent Teams各エージェントの結果を統合エージェントが参照するため
同じプロンプトを複数企業に並列実行Sub-agents独立した調査でP2P通信不要
コードレビュー（セキュリティ・パフォーマンス・スタイル）Agent Teamsレビュアーが互いの指摘を参照して重複を避けるため
並列でテストを実行・修正Sub-agentsテストは独立して実行可能
仕様作成→レビュー→テスト実装の連鎖Agent Teams前工程の成果物を次工程が参照する必要があるため
複数ファイルの並列リファクタリングSub-agentsファイルが独立していればP2P通信は不要
