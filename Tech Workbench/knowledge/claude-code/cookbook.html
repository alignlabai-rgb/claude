Recipe 01: The "Strangler Fig" Refactoring LEGACY CODE

動いている巨大レガシーシステムを止めずに、少しずつ新システムへ移行させる高等テクニック（ストラングラー・フィグ・パターン）の実践。

Step 1: Interface Extraction

まず、既存の複雑な関数をラップするインターフェースを作らせます。

$ claude "src/legacy/OldBilling.js の `calculateTax` メソッドと同じ入出力を持つTypeScriptインターフェース `ITaxCalculator` を定義して。実装はまだしなくていい。"
Step 2: Dual Implementation

新旧両方のロジックを並行稼働させる準備をします。

$ claude "ありがとう。次に `BillingProxy` クラスを作成して。このクラスは `ITaxCalculator` を実装し、内部に `OldBilling` と `NewBilling` (まだ空でいい) の両方を持つ。 featureFlag が true なら New を、false なら Old を呼ぶようにして。"
Step 3: Test & Switch

新しいロジックを実装し、テストを通します。

$ claude "完璧だ。では `NewBilling` に最新の税率計算ロジックを実装して。その上で、Old と New が同じ入力を受けた時に、同じ結果を返すかを検証するテスト `parity.spec.ts` を書いて。"
Recipe 02: "Read-Only" Analyst Mode RESEARCH

コードを書き換えずに、プロジェクトの構造や潜在的なバグだけを調査させる、安全な分析モード。

The Prompt (Copy & Paste)
$ claude --print "このプロジェクトを Security Analyst として監査して。 以下の3点に絞ってレポートを出力せよ（ファイル修正は禁止）。 1. **SQL Injection**: 生のSQL文字列結合を行っている箇所はあるか？ 2. **Sensitive Data**: APIキーやパスワードがハードコードされていないか？ 3. **Dependency**: 脆弱性が既知の古いライブラリ (`package.json`) はあるか？ 出力はMarkdown形式のレポート `AUDIT_REPORT.md` として標準出力すること。" > AUDIT_REPORT.md
Recipe 03: The "TDD" Loop BEST PRACTICE

最も堅実な開発スタイル。テストファーストをAIに強制する。

Phase 1: Generative Testing
$ claude "機能: ユーザーがCSVをアップロードし、一括登録するAPI 要件: - 1回の上限は1000件 - メールアドレスの形式チェック必須 - 重複ユーザーはスキップ 上記仕様を満たす `import-users.spec.ts` を作成して。 実装クラス `ImportService` はまだ存在しないので、インポートエラーになっても構わない。 まずは『あるべき仕様』をテストコードとして定義して。"
Phase 2: Implementation to Green
$ claude "テストありがとう。 では、このテスト `import-users.spec.ts` がコンパイルエラーなく実行でき、かつ全てのテストケースが PASS するように `ImportService.ts` を実装して。 必要なライブラリがあれば `npm install` していい。"
