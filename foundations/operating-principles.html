運用原則

このリポジトリを複数の Claude インスタンスが非同期で更新し続けるために必要な原則です。「なぜその原則があるか」を記録することで、新しいインスタンスが初めて来ても同じ判断ができる状態を目指しています。

基本姿勢

壊してよいではなく、戻せる前提で更新する

実験的な運用であっても、破壊的な変更は Git の履歴に必ず残す。削除より「別の場所に移す」を優先する。

→ なぜ：誰かが何かを消したとき、「意図的な削除」と「事故」を区別できないと復元判断ができない。

ルートの index.html を総合ポータルとして維持する

どんな変更をしても、ルートの入口だけは常に全カテゴリへ辿れる状態にする。

→ なぜ：初めてアクセスするインスタンスや人が、ここから全体を把握できることが最低保証。

カテゴリ責務をまたぐ変更は logs/ に記録する

複数カテゴリに影響する構造変更は、変更日・変更内容・理由を logs/ に残す。

→ なぜ：なぜ構造がこうなっているかを後のインスタンスが理解できない場合、同じ変更を繰り返すか、意図に反した変更をする。

コンテンツ追加の原則

追加前にカテゴリを決める

新しいページを作る前に、どのフォルダに属するかを決め、そのフォルダの index から辿れるようにする。

→ なぜ：フォルダに入れてもリンクされていないページは「存在しない」のと同じ。発見できないコンテンツに価値はない。

リンク切れと文字化けを最優先で直す

新規追加よりも既存の壊れた状態の修正を優先する。

→ なぜ：壊れたリンクは信頼性を損なう。「増やしているが使えない」状態は「何もない」より悪い。

次のインスタンスが理解できる状態を目指す

「動く」だけでなく、なぜその構造・内容にしたかが伝わるように書く。

→ なぜ：このリポジトリは複数インスタンスが引き継いで更新する。意図が伝わらなければ毎回ゼロから判断することになる。

インスタンス間の協調原則

このリポジトリは複数の Claude インスタンスが同時・非同期で更新する場合があります。

ファイル単位で責務を分ける

同一ファイルへの同時編集は Git コンフリクトの原因になる。担当領域を分けて作業する。

CLAUDE.md を共通の判断基準とする

迷ったときは CLAUDE.md に立ち返る。CLAUDE.md に書かれていない判断をしたときは、その判断を logs/ か foundations/ に記録する。

main への直接 push は小さな変更のみ

大きな構造変更はブランチを切って PR 経由でマージする。小さな修正・追記は main への直接 push で可。

判断に迷ったときのフロー

  1. CLAUDE.md を確認する
  2. foundations/ の関連ページを確認する
  3. logs/ の最近の記録を確認する
  4. それでも迷う場合は、より保守的な選択をして logs/ にその旨を記録する
  5. 大きな変更は人間（リポジトリオーナー）に確認を求める
