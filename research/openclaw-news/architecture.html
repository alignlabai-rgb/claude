2. コアコンポーネント

📨 メッセージングレイヤー

ユーザーからの入力を受け取り、内部メッセージ形式に変換するアダプターパターンで実装。各メッセージングアプリごとにアダプターが存在します。

  • signal-adapter： signal-cliをバックエンドに使用。E2E暗号化を維持
  • telegram-adapter： Bot API経由。Webhookモードとポーリングモードに対応
  • discord-adapter： Discord.js v14を使用。スラッシュコマンドに対応
  • api-adapter： REST API経由のプログラマティックアクセス
🧠 コアエンジン

OpenClawの中核。タスクの解析、計画立案、実行管理を担当します。

  • タスクプランナー： ユーザーの指示を分析し、実行計画を立案
  • Skillルーター： 適切なSkillを選択し、実行順序を決定
  • コンテキストマネージャー： 会話履歴とタスク状態を管理
  • エラーハンドラー： 失敗時のリトライ・フォールバック処理
⚙️ Skillランタイム

Skillの読み込み・実行・ライフサイクル管理を行うサンドボックス環境。

  • Skillローダー： SKILL.mdを解析し、メタデータと実行手順を抽出
  • サンドボックス： 各Skillを隔離された環境で実行（セキュリティ対策）
  • 依存関係リゾルバー： Skill間の依存関係を解決し、正しい順序で実行
  • ツールプロキシ： 外部サービス（メール、カレンダー等）へのアクセスを仲介
🤖 LLMプロバイダーレイヤー

複数のLLMプロバイダーを統一的なインターフェースで抽象化。

  • プロバイダーアダプター： 各LLM APIの差異を吸収するアダプター群
  • ルーティングエンジン： タスク特性に応じて最適なLLMを自動選択
  • キャッシュレイヤー： 同一クエリの結果をキャッシュしてコスト削減
  • フォールバック： プライマリLLMが利用不可の場合のバックアップ設定

3. データフロー

🔄 タスク実行の流れ
  • 1. 入力受信： メッセージングアダプターがユーザー入力を受信
  • 2. 前処理： 入力を正規化し、プロンプトインジェクション検出を実行
  • 3. 計画立案： コアエンジンがLLMを使って実行計画を生成
  • 4. Skill選択： Skillルーターが適切なSkillを選択
  • 5. 実行： Skillランタイムが計画に沿ってSkillを順次実行
  • 6. 結果統合： 各Skillの出力を統合して最終結果を生成
  • 7. 出力配信： メッセージングアダプターを通じてユーザーに結果を返送

4. ディレクトリ構造

~/.openclaw/
├── config/
│   ├── settings.json        # 全体設定
│   ├── providers.json       # LLMプロバイダー設定
│   └── messaging.json       # メッセージング設定
├── skills/
│   ├── bundled/             # バンドルSkill
│   ├── global/              # グローバルSkill
│   └── workspace/           # ワークスペースSkill
├── data/
│   ├── context.db           # コンテキスト・会話履歴
│   ├── cache/               # LLMレスポンスキャッシュ
│   └── tasks/               # タスク実行ログ
├── logs/
│   ├── app.log              # アプリケーションログ
│   ├── security.log         # セキュリティログ
│   └── audit.log            # 監査ログ
└── bin/
    └── openclaw             # 実行ファイル

5. 技術スタック

  • 言語： TypeScript（コアエンジン）、Python（一部Skill）
  • ランタイム： Node.js 18+
  • データベース： SQLite（ローカル）、PostgreSQL（エンタープライズ）
  • メッセージキュー： BullMQ（Redis）
  • API フレームワーク： Fastify
  • WebSocket： ws ライブラリ
