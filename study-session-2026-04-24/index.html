最初に結論

Claude Code は 「質問に答えるAI」ではなく、「目標を受けて作業を進めるAI」 である。

ChatGPTのようなチャット型AIは、基本的に「呼ばれたら答える」仕組みです。 一方でClaude Codeは、必要な資料や社内外の情報、作業環境につながりながら、 調査、整理、資料作成、確認を一連の作業として前に進めることができます。

Claude Codeとは何か

定義

Claude Code は、必要な資料や道具につながりながら、自律的に作業を進められる汎用AIエージェントです。

本質

価値は「答えの賢さ」だけではなく、人の代わりに作業を進められることにあります。

誤解しやすい点

普通のチャットAIや単なるCopilotと同じだと思うと、本質を見誤ります。

Claudeでできることの全体像

通常のClaude

相談、壁打ち、要約、文章の下書き。まず考えを整理したいときの入口として使いやすい。

Claude Cowork

非技術者でも扱いやすい画面で、長めの調査、資料整理、文書作成をまとめて任せやすい。M&Aチームには入口として相性が良い。

Claude Code

調査、整理、資料化、確認をまとめて前に進める中心機能。企画チームで一番効きやすいのはここです。

定時実行・接続

決まった時間に動かす、外部情報を取りに行く、通知するなど、運用として回す仕事に向く。

Claude Design

図、スライド、1枚資料の見せ方を早く整える用途。投資仮説や案件比較の中身を作った後、経営説明用の形に落とす場面で効く。

共同作業

チームでの受け渡し、依頼の分担、共同整理のような場面で効きます。

画面操作

人がブラウザやPCで行う操作に近いことを、AIが代わりに進める方向も見えてきています。

将来の拡張

権限やセキュリティ設計次第で、社内外システムとの連携や自動化の幅が広がります。

通常のClaude / Claude Cowork / Claude Code の使い分け

通常のClaude

会話しながら考えを整理する入口。論点出し、壁打ち、要約、たたき台の方向づけに向く。

Claude Cowork

長めの調査や資料整理を、画面上で任せやすい入口。専門的な操作を意識せずに、複数のファイルをまたぐ作業をまとめて進めたいときに向く。

Claude Code

より複雑な作業、接続、定時実行、作り込み、自動化まで踏み込みたいときの本体。深く使い込むほど差が出る。

M&Aチームにとっては、使い始めやすさでは Claude Cowork、複雑な調査や自動化まで含めた実行力では Claude Code、という整理が分かりやすいです。

M&A・投資検討での具体ユースケース

初期調査パック

候補企業の会社概要、事業、主要ニュース、財務概況、株主、規制論点、想定シナジー、追加確認事項をまとめる。

比較メモ

複数候補を同じ項目で並べ、投資仮説や優先順位の検討材料を揃える。

経営説明メモ

1ページ要約、主要リスク、次に聞くべき論点を短時間で叩き台化する。

定点モニタリング

重点国、重点業界、重点企業のニュースや開示を定時で収集し、差分だけを通知する。

DD前整理

財務、事業、法務、規制、コンプライアンスの観点を初期段階で洗い出して、抜け漏れを減らす。

資料再構成

既存メモや社内資料を読み込み、説明相手ごとの粒度に合わせて組み替える。

案件発生時の初動セット

候補企業名を入れると、会社概要、主要ニュース、最近の開示、論点、追加確認事項まで最初の1枚を短時間で立ち上げる。

公開事例から見ても近い使い方

初期スクリーニング用メモ

Anthropic の金融向け公開チュートリアルでは、公開情報から投資メモを作り、初期スクリーニングや委員会向けの叩き台にする流れが示されています。企画チームで言えば、候補企業の一次整理にかなり近いです。

比較プレゼンの作成

同じく公開チュートリアルでは、複数企業の比較指標を集めて、比較プレゼンまでまとめる流れが示されています。候補先の優先順位づけや、上申前の論点整理に直結します。

経営向け説明の下準備

IG Group の公開事例では、Claude を使って戦略プロジェクトを分析し、経営層向けの説明まで短時間で組み立てる流れが紹介されています。企画メモを役員説明用の形に整える用途と相性が良いです。

多ソース・多言語の調査圧縮

NBIM や金融向け公開資料では、調査レポート、市場データ、規制文書、多言語ニュースの横断処理が強調されています。海外案件やクロスボーダー投資の初期調査で効きやすい領域です。

DD文書の自動処理

BlueFlame の公開事例では、データルーム文書の自動処理、企業比較、KPI抽出、デューデリジェンス自動化が示されています。少人数でも分析の厚みを上げられる、という点でM&Aチームに近いです。

文書分析と面談準備の圧縮

Chronograph の公開事例では、顧客調査、文書分析、面談準備のような反復作業をClaudeで圧縮したと紹介されています。案件面談前の下調べや論点整理の短縮に近い使い方です。

要するに、世の中で見えている使い方も「最終判断の自動化」ではなく、「初期調査、比較、説明準備を圧縮する」方向が中心です。企画チームの実務とかなり重なります。

チャットAIと何が違うのか

チャット型AI
  • 質問されると答える
  • 1回ごとの応答が中心
  • 人が毎回次の指示を出す
  • 人が画面の前にいないと進まない
  • 主役は人、AIは補助
Claude Code
  • 目標を受けて、手順を組み立てる
  • 複数ステップの作業をまとめて進める
  • 必要に応じて調査、編集、実行、修正を繰り返す
  • 人はずっと付きっきりでなくてよい
  • 主役は作業の進行、AIが実行主体に近い

Copilotとの違い

観点 Copilot型 Claude Code型
基本姿勢 人が運転し、AIが補助する 人が目標を渡し、AIがかなりの部分を進める
単位 補完、提案、回答 調査、整理、資料化、確認、報告
人の関与 継続的に必要 開始時と確認時が中心
向いている場面 その場の作業支援 まとまった作業の委任

Copilotでは足りにくい理由

Copilotが向いていること
  • 目の前の文書やコードの補完
  • その場での提案や修正候補
  • 人が操作主体で進める作業
企画チームで不足しやすい点
  • 案件横断の調査をまとめて走らせにくい
  • 定時監視や定期配信のような運用に組み込みにくい
  • 人が逐次ハンドリングする前提が強い
  • 「調べる、整理する、まとめる、報告する」を一気通貫に回しにくい

企画チームで必要なのは、文書補完の道具よりも、案件調査や定点監視を一連で前に進めるエージェントです。Claude Code はこの差が大きいです。

AIの使い方は次の段階に進んでいる

AI活用は、Prompt から Context、そして Harness へ移っている。

ここでいう Harness は、単なるプロンプトや知識ではなく、AIが働くための環境、役割、手順、接続先、確認フローまでを含んだ全体像です。

3段階の整理

Prompt Engineering

AIに何を言うかを設計する段階です。

  • 指示文の書き方
  • 依頼の粒度
  • 役割の指定
Context Engineering

AIに何を見せるかを設計する段階です。

  • 背景情報
  • 既存ファイル
  • 制約条件
Harness Engineering

AIをどんな職場で動かすかを設計する段階です。

  • 使えるツール
  • 役割分担
  • 承認フロー
  • 定期実行

企画チーム向けに言い換えると

チーム運営の見方
  • Prompt = 部下への指示の出し方
  • Context = 部下に渡す資料や背景
  • Harness = 部署設計、権限、業務フロー、使える道具

Claude Codeの本質は「優秀な返答者」ではなく、「組織に組み込める作業者」に近い、という理解になります。

企画業務の見方
  • Prompt = 「この企業を調べて」の指示
  • Context = 過去案件、投資方針、論点メモ
  • Harness = 調査フロー、標準観点、定時監視、通知先

AI活用は「賢い相談相手を置く」話から、「企画業務の一部を継続的に回す」話へ移っています。

この考え方でClaude Codeを見ると

Prompt
最初の指示
Context
資料・コード・制約
Skills
能力の標準化
Harness
チーム・定時実行・接続

Claude Codeは単体で使っても強いですが、本当に差が出るのは Skills、Agent Teams、定時実行、接続と組み合わせて「組織として動く形」にしたときです。

企画チームでClaude Codeが実際にやること

調査

企業情報、ニュース、規制動向、公開資料を横断して、一次整理や比較材料をまとめる。

企業調査 ニュース収集 論点整理
資料化

PowerPoint、Excel、Word に入れる前段のメモ、比較表、説明用の叩き台を作る。

比較表 要約 叩き台
確認

情報の抜け漏れ、観点の偏り、更新差分を見直し、何を追加で確認すべきかを整理する。

抜け漏れ確認 差分把握 次アクション

チャットAIにはない6つの能力

1

ツールを使う

検索、ファイル読込、編集、コマンド実行などを、会話の外まで含めて進められる。

2

複数ステップで考える

1回答えるだけではなく、調査→整理→資料化→確認の往復をまとめて回せる。

3

情報のまとまりを扱う

複数の資料や過去メモ、比較観点の関係性を踏まえて整理できる。

4

並列化する

タスクを分け、複数の観点や作業を同時並行に進めやすい。

5

能力を固定化する

Skills によって、毎回説明しなくても同じ流儀で動かせる。

6

運用に組み込める

定時実行や外部接続と組み合わせることで、人不在でも定期処理や連携ができる。

Claude Codeが強い理由

実務的な見方

Claude Code の強さは、単に会話するだけでなく、資料、ファイル、ツール、各種機能につながりながら作業を前に進められることにあります。

  • 資料やファイルを読む
  • 必要な情報を集める
  • 整理して叩き台を作る
  • 足りない論点を洗い出して報告する
業務イメージ
目的を渡す
  ↓
Claude Codeが公開情報や資料を読む
  ↓
比較表やメモの叩き台を作る
  ↓
抜け漏れや追加論点を整理する
  ↓
人が判断しやすい形で返す

Claude の各機能を業務起点で見ると

通常のClaude

相談、壁打ち、要約、文章の下書きに向きます。アイデア整理や論点の荒出しに使いやすい入口です。

Claude Code

調査、整理、資料化、確認をまとめて前に進める中心機能です。企画チームで一番効きやすいのはここです。

共同作業系の機能

チームでの受け渡し、レビュー、依頼の分担、共同整理のような場面で効きます。

定時実行・接続系の機能

毎朝のニュース要約、重点企業監視、外部情報の取得、通知など、運用として回したい仕事に効きます。

重要なのは「どの機能名か」ではなく、「企画チームのどの仕事に効くか」です。Claude Code はその中心で、他の機能はそれを補強する位置づけです。

Skillsとは何か

プロンプト から、能力ファイル へ。

Skills は「毎回同じことを説明する」のをやめて、AIに持たせたい能力や手順をファイルとして固定化する仕組みです。 これは個人のコツではなく、組織の能力を資産化するための考え方です。

なぜ重要か

能力の資産化

ベテランだけが知っている手順を、AIが再利用できる形に保存できる。

属人化の解消

人によって品質がばらつく仕事を、一定の型に寄せられる。

拡張性

1つ作れば、複数人、複数案件、複数エージェントで使い回せる。

Skillsのイメージ

旧来
  • 毎回長いプロンプトを書く
  • 人によって言い方が違う
  • 品質が安定しない
  • 同じ説明を何度も繰り返す
Skills
  • 役割、手順、注意点を1回定義する
  • 以後はその能力を呼び出すだけでよい
  • チーム全体で同じ型を共有できる
  • 改善すれば全員に反映できる

分析

financial-analyst

法務

legal-reviewer
契約やリスクの見方を標準化する。

business-analyst

報告

ばらばらの調査結果を1枚にまとめる。

1体ではなく、チームとして動かす

Claude Code の価値は単体利用でも十分ありますが、より大きいのは「複数の役割を分担させる」発想です。 これが Sub-agents や Agent Teams の世界です。

Sub-agents と Agent Teams

Sub-agents

独立した作業を並列実行するのに向いています。

  • 調査A / 調査B / 調査Cを同時に回す
  • 複数候補企業の独立調査
  • 複数観点の要約
Agent Teams

役割同士が連携しながら進む、より組織的な形です。

  • 調査担当 → 論点整理担当 → まとめ担当
  • 書き手と確認担当の往復
  • 役割別に専門性を分ける

チーム内でどう説明するか

従来

1人が全部やる。AIは相談相手に留まる。

これから

1人が目標を決め、複数のAIが役割分担して進める。

人の役割

作業者から、設計者・監督者・判断者へ寄っていく。

定時実行・自動実行とは何か

呼ぶAI から、定期的に動くAI へ。

定時実行・自動実行は、人が毎回「やって」と頼まなくても、決まったタイミングやトリガーでAIに仕事をさせる考え方です。 ここまで来ると、AIは道具というより運用の一部になります。

わかりやすい例

ニュース監視

毎朝8時に関連ニュースを収集し、要点だけをまとめて共有する。

重点企業監視

重点企業や重点地域の更新を追い、変化があったときだけ要点を知らせる。

比較メモ更新

定期的に比較表や候補企業メモを見直し、新しい情報だけを追記する。

定点調査

定期的に競合や市場トレンドを見て、差分だけ報告する。

企画チームで現実に考えやすい定時実行

朝の市場・案件ウォッチ

毎朝決まった時間に、重点国・重点企業・金融機関の関連ニュースを収集し、日本語で3行要約して配信する。

  • 過去24時間の主要ニュース
  • M&A、出資、規制、経営陣変更を優先
  • 重要度が高いものだけ別通知
重点企業モニタリング

ウォッチリスト企業に関する更新が出たら、要点と影響仮説をつけてメモ化する。

  • 何が起きたか
  • 投資仮説にどう効くか
  • 追加で見るべき論点
月次比較サマリー

海外金融機関や競合の AI・M&A・デジタル投資動向を月次で比較表にして、経営向けの叩き台を作る。

案件発生トリガー

新しい候補企業が登録されたら、初期調査テンプレートに沿って一次調査パックを自動起動する。

なぜこれが重要か

人が忘れない

忙しいと後回しになりやすい監視や定期調査を、運用として固定できる。

情報の鮮度が上がる

必要な情報が毎回同じ形式で集まり、比較しやすくなる。

人は判断に寄れる

収集と整理に使う時間を減らし、論点設定や意思決定準備に集中できる。

定時実行・自動実行の意味

省力化

毎回人が思い出して依頼する必要がなくなる。

抜け漏れ防止

「忙しくてやらなかった」を減らせる。

24時間化

人が見ていない時間にも前処理や監視を進められる。

接続と拡張の話

賢いAI を、使えるAI に変えるのが接続です。

AI単体では、知っていることを答えるだけです。接続や拡張の考え方を使うと、 社内外のツール、データ、ワークフローとつながり、実務の中で役に立つようになります。

接続すると何が変わるか

情報源

ファイル、GitHub、カレンダー、DB、社内システムなどから直接材料を取れる。

行動

読むだけでなく、投稿、更新、記録、通知、必要に応じた自動操作まで進められる。

再現性

特定の人だけができる操作を、型として運用に組み込みやすくなる。

組み合わせると何が起きるか

Skills = 能力を固定する
Agent Teams = 役割を分ける
定時実行 = 定期的に起動する
接続 = 必要な道具と情報につなぐ

企画チームで考えやすい接続先

情報取得

ニュース、公開開示、社内ファイル、比較メモ、ウォッチリストなどを読み込む。

通知

メール、チャット、共有先への通知やメモ出力につなげる。

将来の拡張

セキュリティや権限次第で、より広い社内外システムとの連携も視野に入る。

すべてを最初からつなぐ必要はありません。まずは情報取得と通知のような低リスク領域から始めるのが現実的です。

Computer Useとは何か

AIが答える だけでなく、画面を使って作業する 方向へ進んでいる。

Computer Use は、AIが人の代わりにPCやブラウザを操作する方向性です。まだ何でも完全自動でできる、という話ではありませんが、 企画チームの実務で言えば、人がブラウザで行っている検索、閲覧、照合、転記のような前工程をAIが肩代わりしていくイメージとして捉えると分かりやすいです。

企画チームでイメージしやすい使い方

候補企業の初期調査

企業サイト、ニュース、開示、業界記事を横断して見ながら、会社概要、事業、最近の動き、気になる論点を拾い上げる。

複数ソースの照合

複数画面をまたいで数値や記述を確認し、差分や不一致、追加で確認すべき点を洗い出す。

定型的な画面作業

人が繰り返している検索、一覧確認、転記、要点抽出のような単純作業をまとめて進める方向性があります。

M&A・投資検討に引きつけた見せ方

見せると伝わりやすい例
  • 候補企業名を起点に、会社情報、主要ニュース、最近の開示を順に確認する
  • 複数サイトを見ながら、事業内容や地域展開の記述を照合する
  • 重点企業の更新情報を見に行き、変化があればメモ化する
  • 人なら15分かかる画面巡回を、短い時間で一通り回す
言い方のポイント
  • 「自動売買」や「意思決定の自動化」とは言わない
  • 「調査の前工程を肩代わりする」と表現する
  • 「人が見に行く画面を先に回ってくれる」と言うと伝わりやすい
  • 最終判断は人、画面作業の前半をAIが受け持つと整理する

誤解しないためのポイント

できること
  • 人の画面操作に近い流れを再現する
  • ブラウザベースの情報収集を補助する
  • 調査作業の入口を自動化する
注意点
  • 権限やセキュリティの制約を受ける
  • 重要判断をそのまま任せるものではない
  • 最初から全部を置き換える前提ではない

見せ方としては、「候補企業の調査や重点先のモニタリングで、人が最初にやる画面巡回をAIが先に進める方向がある」と説明するのが自然です。

導入の考え方

Claude Code はすぐに全てを自動化するものではありません。まずは低リスクで反復が多く、価値が見えやすいところから始めるのが現実的です。

導入前に押さえる前提

期待すべきこと
  • 初期調査や整理の速度を上げる
  • 比較や監視の抜け漏れを減らす
  • 人が考えるための叩き台を早く作る
  • 反復業務を定期実行に載せる
期待しすぎないこと
  • 投資判断そのものを任せるものではない
  • 機密情報を無条件に入れてよいわけではない
  • 事実確認や最終判断は人が持つ
  • 最初から全部の業務を置き換えるものではない

導入の本質は、企画チームの判断をAIに渡すことではなく、調査と整理の前工程を厚くして、人が判断に集中できる状態を作ることです。

現実的な導入モデル

基本方針
  • 今ある業務PCに、そのまま全面開放する前提では進めない
  • AI利用に適した専用PC・専用環境を切り出して整備する
  • セキュリティチームと詰めながら、使える範囲を段階的に広げる
  • まずは調査、整理、監視などの前工程から実務に入れる
想定する制御
  • 外部メール送信や外部共有は最初から自由にしない
  • OneDrive や社内データへは直結ではなく、必要に応じて中間領域を置く
  • 権限、接続先、保存先、操作範囲を最初に絞る
  • 万一の誤作動時でも影響を閉じ込める設計を優先する

ポイントは「高機能なAIをそのまま社内に入れる」ことではなく、「業務に使えるが、万一でも被害が広がりにくい環境を先に作る」ことです。

セキュリティチームと詰める論点

端末

AI用の専用PCにするのか、VDIや隔離環境にするのか。通常業務端末とは役割を分けて考える。

ネットワーク

どこへ接続を許すか、何を遮断するか、ログをどう残すか。調査用途と外部送信は分けて考える。

データ

どのデータを直接扱えるか、中間保管領域を置くか、機密度に応じて何を入れないかを整理する。

なぜ専用環境が必要か

そのまま入れると難しい理由
  • Computer Use のような機能は、理論上できる範囲が広い
  • プロンプトインジェクションや誤操作時の影響を小さくする必要がある
  • 通常端末の権限とAIの行動範囲が直結すると、説明責任が重くなる
  • 社内展開前に、安全な成功例を作る必要がある
専用環境で得られること
  • 接続先と操作範囲を限定したまま実務で試せる
  • 監視、ログ、運用ルールを前提にした導入ができる
  • 業務で使える範囲を、実績ベースで少しずつ広げられる
  • 他部署へ展開するときの実例として使える

企画チームで最初に始めるとよい領域

始めやすいもの
  • ニュース監視と定時要約
  • 候補企業の初期調査パック
  • 比較表の叩き台作成
  • 社内説明メモの一次ドラフト
後から広げるもの
  • 複数観点の並列調査
  • Skills によるチーム標準化
  • 定期実行と通知連携
  • 案件発生時の半自動ワークフロー

プロダクトは一つに固定しない

考え方

最初の実例として Claude Code を軸に見るのは自然ですが、将来的には特定の製品名に固定せず、セキュアなAI実行環境の中で複数の選択肢を評価していく考え方が現実的です。

意味

重要なのは製品名そのものではなく、企画チームの実務に役立つこと、セキュリティ条件を満たすこと、運用可能であることです。

導入ステップ

1

単発タスクで試す

ニュース要約、初期調査メモ、比較表の叩き台など、戻しやすく価値が見えやすいものから始める。

2

毎回同じ説明をしているなら、それは Skill 候補です。

3

役割を分けて並列化する

調査、比較、要約など、独立しやすい作業を分ける。

4

定時実行と接続を加える

定期的な仕事や通知のように、人手を使わなくてよいところから運用に組み込む。

説明用の言い回し

短い説明

Claude Codeは、チャットで答えるAIではなく、資料や情報源、作業環境につながりながら仕事を前に進めるAIエージェントです。

マネジメント向け

これは「AIが賢くなった」というより、「企画チームの調査、整理、定点監視の一部を運用として回せるようになった」という変化です。人は実行者から設計者・判断者へ寄ります。

AIに任せること / 人が判断すること

AIに任せやすいこと
  • 公開情報の収集と一次整理
  • 比較表、要約、叩き台の作成
  • 重点企業や重点地域の継続監視
  • 論点や追加確認事項の洗い出し
人が判断すべきこと
  • 投資判断そのもの
  • 交渉方針や対外コミュニケーション
  • 機密情報の扱いと最終責任
  • どの論点を重視するかの優先順位付け

最後のまとめ

答えるAI から、進めるAI へ。
個人の工夫 から、組織の仕組み へ。

これがこのデモで伝えたい中心です。Claude Code は単なるチャットAIの強化版ではありません。 組織の仕事の進め方そのものを変える可能性があるため、Skills、Agent Teams、定時実行、接続のような周辺概念まで含めて理解する必要があります。

その上で実務導入は、素の業務環境にそのまま入れる話ではなく、専用環境、限定接続、段階導入を前提に進めるべきです。 この部で安全に実例を作れれば、銀行全体への展開を考えるうえでの先行事例にもなります。
