Mission Control

Programming the Agent: The Art of the Spec

自律型エージェントに対する指示は、「プロンプト」ではありません。
それは「Spec (仕様書)」であるべきです。
曖昧な Spec は、曖昧な成果物（または無限ループ）を生みます。

The Perfect Mission Spec

OpenClaw に仕事を投げるときは、必ず以下のフォーマットで記述した Markdown ファイル (`MISSION.md`) を渡してください。

Template: MISSION.md
# Mission 001: User Profile Refactoring ## 1. Objective (目的) 現在の `UserProfile` コンポーネントは巨大すぎる。 これを `Avatar`, `Bio`, `Stats` の3つのサブコンポーネントに分割し、可読性を向上させること。 ## 2. Constraints (制約条件) ← 最重要 - **No Logic Change**: ビジネスロジックは一切変更してはならない。 - **Library Lock**: 新しいライブラリ (`npm install`) は禁止。 - **UI Preservation**: DOM構造やCSSクラス名は変更しないこと（E2Eテストが壊れるため）。 ## 3. Definition of Done (完了の定義) ← エージェントはここを目指す - [ ] `src/components/profile/` 以下に3つの新ファイルが存在する。 - [ ] `npm run test` が全てPASSする。 - [ ] `npm run lint` がエラーゼロである。 ## 4. Resources - 参照すべきファイル: `src/components/UserProfile.tsx` - 参照してはいけないファイル: `src/legacy/*`
このファイルを保存し、$ openclaw run MISSION.md と実行します。

Why "Definition of Done"?

人間と違い、エージェントは「いい感じになったら終わり」という感覚を持ちません。
明確な終了条件（テストオールグリーン、ファイル存在確認など）を与えないと、彼は「もっと綺麗にできるはずだ」と無限にリファクタリングを続けてしまいます。

Intervention (介入)

Mission実行中、Operatorはログ監視画面（Mission Control）で状況を見守ります。

  • Pause & Redirect: 「方向性が違う」と思ったら即座に一時停止し、チャットで軌道修正を指示します。
  • Inject Knowledge: エージェントが迷っていたら、ヒントや追加資料を途中で差し入れます。

完全に放置するのではなく、「困っているときだけ助け船を出す」のが熟練Operatorの技です。
